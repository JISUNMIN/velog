<p><strong>서드파티(Third Party)</strong>는  
<strong>우리 회사/팀이 직접 만들지 않고, 외부에서 제공되는 모든 기술 요소</strong>를 의미합니다.</p>
<blockquote>
<p>“내가 만든 게 아닌 모든 외부 기술 요소 = 서드파티”</p>
</blockquote>
<p>서드파티는 아래와 같은 외부 기술들을 모두 포함하는 상위 개념입니다:</p>
<ul>
<li><strong>서드파티 라이브러리</strong>  </li>
<li><strong>서드파티 SDK</strong>  </li>
<li><strong>서드파티 API</strong>  </li>
<li><strong>서드파티 서비스(클라우드 등)</strong>  <p align="center">
<img src="https://velog.velcdn.com/images/sunmins/post/81da0d0e-d3a4-49f1-bd15-89457f0288a4/image.png" width="600" />
</p>


</li>
</ul>
<hr />
<h2 id="💡-왜-서드파티를-사용할까">💡 왜 ‘서드파티’를 사용할까?</h2>
<p>서비스를 만들 때 필요한 기능을 <strong>전부 직접 만들 필요는 없습니다.</strong><br />날짜 파싱, 로그인, 지도, 결제, 인증 등 많은 기능들은 이미 다른 회사에서<br />검증된 형태로 제공하고 있습니다.</p>
<p>서드파티를 사용하면:</p>
<ul>
<li>복잡한 기능을 빠르게 붙일 수 있고  </li>
<li>검증된 기술을 안정적으로 활용할 수 있으며  </li>
<li>개발 시간과 비용을 크게 줄일 수 있습니다.</li>
</ul>
<p>특히 <strong>React Native, Flutter 같은 하이브리드 앱</strong>에서는<br />네이티브 기능을 서드파티 라이브러리나 SDK로 구현하는 경우가 매우 많습니다.</p>
<p>또한 중요한 점은:</p>
<blockquote>
<p><strong>서드파티는 라이브러리뿐 아니라 SDK·API·클라우드 서비스까지 모두 포함하는 넓은 개념이다.</strong></p>
</blockquote>
<hr />
<h2 id="🔍-서드파티의-특징">🔍 서드파티의 특징</h2>
<h3 id="표준-기능을-빠르게-붙일-수-있다">표준 기능을 빠르게 붙일 수 있다</h3>
<ul>
<li>인증(OAuth)  </li>
<li>결제(PG사)  </li>
<li>지도 엔진  </li>
<li>파일 업로드<br />같은 기능은 직접 만들기 어렵기 때문에<br />서드파티를 쓰는 것이 일반적입니다.</li>
</ul>
<h3 id="검증된-기능을-가져다-쓴다">검증된 기능을 가져다 쓴다</h3>
<p>내가 직접 만든 기능보다<br />외부 기업이 수년간 유지보수한 기능이 더 안정적일 때가 많습니다.</p>
<h3 id="서비스-개발-속도가-크게-빨라진다">서비스 개발 속도가 크게 빨라진다</h3>
<p>필요한 기능을 직접 구현하면 몇 주~몇 달이 걸리지만<br />서드파티를 사용하면 <strong>몇 시간 안에 붙일 수 있는 경우도 많습니다.</strong></p>
<hr />

<hr />
<h2 id="⚠-서드파티-의존이-왜-위험할까">⚠ 서드파티 의존이 왜 위험할까?</h2>
<p><strong>개발자 기술면접 노트 – 이남희 지음</strong>에서 언급된 내용입니다.</p>
<blockquote>
<p><strong>서비스의 핵심 기능이 서드파티에 100% 의존하면<br />서드파티가 바뀌거나 장애가 날 때 제품 전체가 흔들린다.</strong></p>
</blockquote>
<p>예를 들어:</p>
<ul>
<li>라이브러리 deprecated  </li>
<li>SDK 강제 업데이트  </li>
<li>API 정책 변경  </li>
<li>외부 서비스 장애(Firebase, AWS 등)  </li>
<li>요금제 변경  </li>
<li>유지보수 종료</li>
</ul>
<p>이런 상황에서는:</p>
<ul>
<li>문제를 내가 고칠 수 없음  </li>
<li>서비스 품질이 외부 회사에 좌우됨  </li>
<li>핵심 로직을 직접 구현해볼 기회가 줄어 성장 범위가 좁아짐  </li>
<li>“내 실력”이 아니라 “서드파티 사용법”만 알게 됨  </li>
</ul>
<p>즉, 서드파티 의존은<br /><strong>서비스 안정성 리스크</strong>와 <strong>개발자 성장 리스크</strong>가 동시에 존재한다는 의미입니다.<br />그래서 주니어일수록 서드파티 의존 프로젝트만 맡으면<br />장기적으로 성장 폭이 제한될 수 있습니다.</p>
<hr />
<h2 id="🤔-서드파티를-언제-쓰고-언제-직접-만들어야-할까">🤔 서드파티를 언제 쓰고, 언제 직접 만들어야 할까?</h2>
<h3 id="서드파티를-써야-하는-기능-직접-구현-불가-or-위험">서드파티를 써야 하는 기능 (직접 구현 불가 or 위험)</h3>
<ul>
<li>OAuth 소셜 로그인 (카카오/구글/애플)  </li>
<li>지도 엔진 전체  </li>
<li>결제 PG 연동  </li>
<li>OS 센서·카메라 기능  </li>
<li>보안·암호화 프로토콜</li>
</ul>
<p>➡ <strong>보안·규제·기술 난이도 때문에 직접 구현하면 안 되는 영역</strong></p>
<hr />
<h3 id="서드파티를-써도-되지만-직접-만들어보면-좋은-기능">서드파티를 써도 되지만, 직접 만들어보면 좋은 기능</h3>
<ul>
<li>자체 로그인(JWT)  </li>
<li>파일 업로드(S3 presigned URL)  </li>
<li>WebSocket 기반 채팅  </li>
<li>푸시 알림 서버  </li>
<li>필터/정렬/검색  </li>
<li>비즈니스 로직 구현</li>
</ul>
<p>이런 것들은 직접 구현해보면 <strong>기본기가 대폭 성장</strong>할 수 있습니다. 
→ 실무에서는 SDK/API를 쓰지만, 학습용으로 직접 구현해보면 매우 좋음.</p>
<hr />
<h3 id="✔-서드파티에만-의존하면-안-되는-핵심-영역">✔ 서드파티에만 의존하면 안 되는 핵심 영역</h3>
<ul>
<li>서비스의 고유 기능  </li>
<li>비즈니스 규칙  </li>
<li>데이터 로직  </li>
<li>주요 알고리즘  </li>
<li>도메인 모델링</li>
</ul>
<p>➡ <strong>기업의 경쟁력이므로 개발자가 직접 만들어야 하는 영역</strong> 
외부에 맡기면 회사 서비스가 “외부 회사의 기능”처럼 변해버립니다.</p>
