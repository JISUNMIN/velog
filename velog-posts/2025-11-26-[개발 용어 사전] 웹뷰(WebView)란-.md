<p><strong>웹뷰(WebView)</strong>는 모바일 앱 안에서 웹사이트를 띄우는 작은 브라우저입니다.
앱에서 특정 버튼을 눌렀을 때, 새로운 앱 화면이 아니라 웹페이지가 앱 내부 창에서 열리는 경우가 있습니다.
그 화면이 바로 웹뷰입니다.</p>
<blockquote>
<p>“앱 안에 심어놓은 미니 웹브라우저”</p>
</blockquote>
<p>라고 생각할 수 있습니다.</p>
<hr />
<h2 id="💡-왜-웹뷰를-사용할까">💡 왜 웹뷰를 사용할까?</h2>
<h3 id="1-앱-업데이트-없이-화면-수정-가능">1) 앱 업데이트 없이 화면 수정 가능</h3>
<p>앱 화면을 전부 네이티브로 만들면,<br />UI 하나만 바꿔도 앱스토어에 다시 빌드/배포해야 합니다.</p>
<p>하지만 웹뷰를 쓰면 <strong>웹으로 만들어진 화면을 앱 안에서 그대로 불러오기만 하면 되기 때문에</strong><br />앱 업데이트 없이도 서버에서 UI를 바로 수정할 수 있습니다.</p>
<p>→ <strong>운영·개발 비용 절감 효과</strong></p>
<hr />
<h3 id="2-하이브리드-앱에서-필수-요소">2) 하이브리드 앱에서 필수 요소</h3>
<p>React Native, Flutter 같은 하이브리드 앱에서는<br />네이티브 화면 + 웹뷰 화면을 섞어서 쓰는 경우가 많습니다.</p>
<p>예:</p>
<ul>
<li>설정/약관 페이지  </li>
<li>이벤트 랜딩 페이지  </li>
<li>결제 페이지(일부 PG사 웹 기반 인증)</li>
</ul>
<p>이런 페이지들은 <strong>변경이 잦고 즉시 반영이 필요하거나</strong>,  
<strong>원본 데이터가 웹 기반(PG사/약관/이벤트 페이지 등)</strong>인 경우가 많습니다.<br />그래서 네이티브로 직접 구현하기보다 <strong>웹뷰로 불러오는 것이 훨씬 효율적</strong>입니다.</p>
<hr />
<h3 id="3-앱에서만-사용">3) 앱에서만 사용</h3>
<p>웹뷰는 <em>앱 안에서 웹을 띄우는 기술</em>이기 때문에<br /><strong>웹(브라우저) 자체에서는 사용되지 않습니다.</strong></p>
<ul>
<li>“앱 → 웹뷰(웹 화면)”은 가능  </li>
<li>“웹 → 웹뷰”는 개념적으로 존재하지 않음</li>
</ul>
<hr />
<h2 id="📱-웹뷰-예시">📱 웹뷰 예시</h2>
<h3 id="✔️-카카오톡-링크-클릭-시">✔️ 카카오톡 링크 클릭 시</h3>
<p>카톡 안에서 웹사이트가 새 창으로 열릴 때 → <strong>웹뷰</strong></p>
<h3 id="✔️-쇼핑-앱-결제-화면">✔️ 쇼핑 앱 결제 화면</h3>
<p>카드사 인증 페이지가 앱 안 작은 브라우저 형태로 열림 → <strong>웹뷰</strong></p>
<h3 id="✔️-서비스-약관-보기">✔️ 서비스 약관 보기</h3>
<p>약관처럼 자주 바뀌는 문서는 웹뷰로 불러와 관리하는 경우가 많음</p>
<hr />
<h2 id="웹뷰와-csrssrisr은-어떤-관계일까">웹뷰와 CSR/SSR/ISR은 어떤 관계일까?</h2>
<p>웹뷰는 단순히 <strong>앱 안에 들어있는 브라우저</strong>입니다.<br />따라서 웹뷰가 어떤 페이지를 띄우든, 그 페이지가 어떤 방식으로 만들어졌는지와는 별개로 동작합니다.</p>
<p>간단하게 정리하면:</p>
<ul>
<li><strong>HTML/JS/CSS 파일 자체는 어떤 방식이든 결국 서버에서 내려온다.</strong>  </li>
<li>그 파일을 “어디서 렌더링(화면으로 그리느냐)”에 따라 방식이 나뉜다.</li>
</ul>
<h3 id="✔️-csr-client-side-rendering">✔️ CSR (Client Side Rendering)</h3>
<ul>
<li>서버에서 JS 파일을 받아오고  </li>
<li><strong>브라우저(웹뷰)가 JS로 화면을 그림</strong></li>
</ul>
<h3 id="✔️-ssr-server-side-rendering">✔️ SSR (Server Side Rendering)</h3>
<ul>
<li>서버가 HTML을 완성해서 보내주고  </li>
<li>웹뷰는 그대로 표시함 → <strong>서버가 그림</strong></li>
</ul>
<h3 id="✔️-isr--ssg">✔️ ISR / SSG</h3>
<ul>
<li>서버가 HTML을 미리 만들어두고  </li>
<li>요청할 때 그 HTML만 보내줌</li>
</ul>
<p>즉,<br /><strong>웹뷰는 렌더링 방식과 무관하게 “결과를 받아서 보여주는 역할”만 한다는 것이 핵심입니다.</strong></p>
<hr />
<h2 id="📎-웹뷰에-들어가는-url은-어떻게-만들까">📎 웹뷰에 들어가는 URL은 어떻게 만들까?</h2>
<p>웹뷰는 기본적으로 <strong>URL을 받아 그 웹페이지를 앱 안에서 띄우는 방식</strong>입니다.<br />따라서 웹뷰에 넣을 페이지가 존재하려면, <strong>해당 웹 페이지가 어디엔가 배포되어 있어야 합니다.</strong></p>
<p>이 배포 위치는 특별히 정해진 것이 없으며, 자주 쓰는 선택지는 다음과 같습니다:</p>
<ul>
<li><strong>Vercel</strong><br />→ 가장 간단하고 자동 배포가 쉬움  </li>
<li><strong>AWS S3 + CloudFront</strong><br />→ 정적 파일 기반, 기업 서비스에서 자주 사용  </li>
<li><strong>Netlify / 자체 서버 등</strong><br />→ 어느 곳이든 URL만 존재하면 웹뷰에서 그대로 사용할 수 있음</li>
</ul>
<p>즉,  </p>
<blockquote>
<p><strong>“웹뷰 = URL을 앱 안에서 띄우는 것”이기 때문에,<br />어떤 방식으로든 웹 페이지가 서버에 배포되어 있어야 URL을 통해 웹뷰로 표시할 수 있습니다.”</strong></p>
</blockquote>
