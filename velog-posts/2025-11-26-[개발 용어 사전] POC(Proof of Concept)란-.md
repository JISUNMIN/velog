<p><strong>POC(Proof of Concept)</strong>는  
<strong>“어떤 아이디어나 기술이 실제로 가능하다는 것을 짧고 빠르게 검증해보는 과정 또는 결과물”</strong>을 의미합니다.</p>
<p>정식 제품을 만드는 것이 아니라,<br /><strong>“이 방법이 되는지 안 되는지 먼저 확인해보자”</strong>라는 목적에 최적화된 실험 단계입니다.</p>
<hr />
<h2 id="💡-왜-poc를-할까">💡 왜 POC를 할까?</h2>
<h3 id="1-기술적으로-가능한지-먼저-확인하기-위해">1) 기술적으로 가능한지 먼저 확인하기 위해</h3>
<p>새로운 기능이나 기술을 도입할 때,<br /><strong>시간·비용을 크게 들이기 전에 실제로 구현이 가능한지</strong> 먼저 실험합니다.</p>
<p>예:</p>
<ul>
<li>새로운 API 연동이 실제로 동작하는지  </li>
<li>특정 AI 모델이 우리 비즈니스 상황에서 성능이 나오는지  </li>
<li>특정 프레임워크(React Native, Flutter)가 우리 요구사항을 충족하는지</li>
</ul>
<p>정식 개발 전에 위험요소를 빨리 발견할 수 있습니다.</p>
<hr />
<h3 id="2-큰-리스크를-줄이기-위해">2) 큰 리스크를 줄이기 위해</h3>
<p>기술 검증 없이 바로 개발에 들어가면:</p>
<ul>
<li>수개월 뒤 “이 방법 안 되네?”가 될 수도 있고  </li>
<li>시간·비용이 대량 낭비될 수도 있음</li>
</ul>
<p>POC는 <strong>“최소 비용으로 리스크를 제거하는 단계”</strong>입니다.</p>
<hr />
<h3 id="3-이해관계자개발팀-경영진-설득용">3) 이해관계자(개발팀, 경영진) 설득용</h3>
<p>POC는 완성품이 아니라 <strong>데모 수준이어도 충분</strong>합니다.</p>
<ul>
<li>“이 기술로 이런 결과가 나옵니다”  </li>
<li>“이 접근법이 실제로 작동합니다”  </li>
<li>“우리 서비스에 적용하면 이런 효과가 있습니다”</li>
</ul>
<p>이렇게 <strong>증거를 보여주기 위한 목적</strong>으로도 많이 쓰입니다.</p>
<hr />
<h2 id="🔍-poc의-특징">🔍 POC의 특징</h2>
<h3 id="가벼움">가벼움</h3>
<ul>
<li>빠르게 만들고, 빠르게 버릴 수 있어야 함  </li>
<li>구조나 디자인은 크게 중요하지 않음</li>
</ul>
<h3 id="실험-중심">실험 중심</h3>
<ul>
<li>품질보다는 “기술적 가능성”이 핵심  </li>
<li>실제 서비스 품질이나 UI/UX는 나중 문제</li>
</ul>
<h3 id="정식-제품이-아님">정식 제품이 아님</h3>
<p>POC는 “작동한다”를 보여주는 수준이며<br />정식 제품(프로덕션 코드)과는 완전히 다름.</p>
<hr />
<h2 id="📱-poc-예시">📱 POC 예시</h2>
<h3 id="✔️-이미지-분석-서비스를-만들고-싶을-때">✔️ 이미지 분석 서비스를 만들고 싶을 때</h3>
<p>오픈소스 모델로 몇 장 테스트해서 정확도 판단 → <strong>POC</strong></p>
<h3 id="✔️-react-native로-gps-기능이-안정적으로-동작하는지-확인">✔️ React Native로 GPS 기능이 안정적으로 동작하는지 확인</h3>
<p>간단한 위치표시 앱을 1~2일 안에 만들어 실험 → <strong>POC</strong></p>
<h3 id="✔️-특정-결제-api가-우리-서비스에서-잘-작동하는지">✔️ 특정 결제 API가 우리 서비스에서 잘 작동하는지</h3>
<p>간단한 테스트 페이지 + 요청/응답 로그 확인 → <strong>POC</strong></p>
<h3 id="✔️-새로운-애니메이션-라이브러리-도입-검토">✔️ 새로운 애니메이션 라이브러리 도입 검토</h3>
<p>샘플 화면 하나 만들어서 성능 체크 → <strong>POC</strong></p>
<hr />
<h2 id="poc-vs-프로토타입-vs-mvp">POC vs 프로토타입 vs MVP</h2>
<ul>
<li><p><strong>POC</strong><br />→ <strong>“이 기능/기술이 실제로 가능한지 확인하는 기술 검증 단계”</strong><br />→ 기능이 된다/안 된다를 빠르게 판단하는 것이 목적</p>
<p>예:</p>
<ul>
<li>특정 라이브러리가 우리 서비스에서 동작하는지 테스트  </li>
<li>외부 API를 연동했을 때 성능이 나오는지 확인  </li>
<li>새로운 기술(예: AI 모델, OCR, GPS 등)이 실사용에 적합한지 검증  </li>
</ul>
<p>즉,  </p>
<blockquote>
<p><strong>POC = 기술·기능이 실제로 가능한지 검증하는 단계</strong></p>
</blockquote>
</li>
<li><p><strong>프로토타입(Prototype)</strong><br /> → <strong>“제품이 어떻게 보이고, 어떻게 동작하는지 빠르게 보여주는 모델”</strong><br />→ UI/UX 흐름, 인터랙션, 화면 구조를 확인하기 위한 목적</p>
<p>예:</p>
<ul>
<li>디자인 시안 기반 Figma 인터랙션  </li>
<li>반만 작동하는 더미 앱 화면  </li>
<li>실제 기능은 없지만 흐름을 보여주는 데모  </li>
</ul>
<p>즉,  </p>
<blockquote>
<p><strong>프로토타입 = 동작 방식·인터랙션·화면 흐름을 예시로 보여주는 단계</strong></p>
</blockquote>
</li>
<li><p><strong>MVP</strong><br />→ <strong>“필수 기능만 넣고 실제 사용자에게 배포하는 최소 제품”</strong><br />→ 시장 반응과 사용자 피드백을 받기 위해 실제로 배포하는 버전</p>
<p>예:</p>
<ul>
<li>핵심 기능 1~2개만 포함된 서비스 첫 출시  </li>
<li>최소한의 로그인 + 핵심 기능만 제공  </li>
<li>실사용 데이터를 수집하기 위한 릴리즈  </li>
</ul>
<p>즉,  </p>
<blockquote>
<p><strong>MVP = 실제 사용자에게 제공하는 1차 출시 버전</strong></p>
</blockquote>
</li>
</ul>
<p> <strong>POC = 기술 검증 (된다/안 된다)</strong><br /> <strong>프로토타입 = 화면·동작 예시 (이렇게 생겼다/이렇게 동작한다)</strong><br /> <strong>MVP = 최소 기능을 갖춘 초기 실제 제품 (일단 출시)</strong></p>
<hr />
<h2 id="📌-한-줄-요약">📌 한 줄 요약</h2>
<blockquote>
<p><strong>POC는 ‘기술적으로 가능한지’를 빠르게 검증하기 위한<br />가벼운 실험 단계 또는 테스트용 결과물이다. 
실제 개발 전 위험 요소를 미리 확인하고,<br />이 방법이 작동한다는 근거를 이해관계자에게 보여주는 용도로도 활용된다.</strong></p>
</blockquote>
