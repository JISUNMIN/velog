## HTTP란?

👉 **HTTP는 웹에서 정보를 주고받기 위한 요청-응답 방식의 통신 규약(프로토콜)**입니다.
브라우저(클라이언트)가 요청을 보내면 서버가 응답을 주는 구조로, 서버가 먼저 클라이언트에게 말을 걸 수는 없습니다.

예시:

* 내가 네이버에 들어가고 싶을 때 → 브라우저가 "페이지 주세요" 요청 → 서버가 "여기 있습니다" 응답.

즉, **단방향(요청 → 응답)** 구조

---

## WebSocket이란?

👉 **WebSocket은 한 번 연결을 맺으면 클라이언트와 서버가 서로 자유롭게 실시간으로 메시지를 주고받을 수 있는 양방향 통신 프로토콜**입니다. 실시간 통신을 위해 사용됩니다.

특징:

* 처음엔** HTTP 요청으로 연결을 열고, 이후엔 WebSocket 통신으로 전환** -> 개발자 도구에서 보면 처음에 HTTP 요청이 보이지만, 연결 후엔 별도의 WebSocket 통신으로 바뀜

![](https://velog.velcdn.com/images/sunmins/post/1b2e2cc9-5c1b-4fbd-88b6-f99e08f3dbda/image.png)

* 서버도 먼저 클라이언트에게 메시지를 보낼 수 있음
* 지연 시간이 짧아 실시간에 적합

예시:

* 카톡 같은 **실시간 채팅**
* 주식, 코인 같은 **실시간 시세 표시**
* 온라인 게임의 **실시간 동기화**
---

### 🤔 서버가 먼저 알려줘야 하는 경우는 어떤게 있을까?

👉 내가** 직접 요청하지 않았는데도 새로고침 없이 화면이 자동으로 바뀌어야 하는 상황**이 이에 해당합니다.

예시:

* 실시간 알림: 메신저에서 상대방이 메시지를 보냈을 때 바로 뜨는 알림
* 실시간 데이터 업데이트: 주식/코인 시세, 스포츠 경기 점수판
* 멀티플레이 게임: 다른 사용자의 움직임이 내 화면에 바로 반영됨
* 실시간 협업: 구글 Docs 같이 누군가 수정하면 내 화면에도 즉시 반영됨

## HTTP vs WebSocket 차이

| 구분    | HTTP               | WebSocket           |
| ----- | ------------------ | ------------------- |
| 통신 방식 | 요청 → 응답 (단방향)      | 연결 후 양방향 자유롭게       |
| 연결 유지 | 요청마다 새로 연결         | 연결 유지 (지속적)         |
| 서버 푸시 | 불가능 (추가 기술 필요)     | 가능 (서버→클라 먼저 전송 OK) |
| 활용 사례 | 웹 페이지 불러오기, API 호출 | 채팅, 실시간 알림, 게임      |

---

## 📝 정리

* **HTTP** → 기본 웹 페이지, 일반 API 통신에 적합 (단순 요청-응답)
⚠️ 실시간 데이터를 만들려면 HTTP로도 가능하지만, 몇 초마다 요청(**Polling**)해야 하기 때문에 비효율적
* **WebSocket** → 실시간 상호작용이 필요한 경우 (채팅, 알림, 게임)
연결을 유지하며 양방향으로 자유롭게 메시지 전송 가능

> Polling이란?
**Polling(폴링)**은 클라이언트가 주기적으로 서버에 요청(API 요청)을 보내서 새로운 데이터가 있는지 확인하는 방식입니다. 쉽게 말하면 “몇 초마다 서버에게 ‘새로운 거 있어요?’ 물어보는것 입니다.
서버가 먼저 데이터를 보내는 게 아니라, 클라이언트가 계속 요청을 보내야 함

👉 **"HTTP는 요청-응답용, WebSocket은 실시간 양방향 통신용"**
