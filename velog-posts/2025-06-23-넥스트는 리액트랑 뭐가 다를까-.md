## 🤔 왜 Next.js를 써야 할까?



> **React는 오직 "화면(UI)"만 만들 수 있는 라이브러리이고**,  
> **Next.js는 화면 + 서버 기능까지 포함한 프레임워크라는 점.**

즉, Next.js는 `Node.js 기반`으로 돌아가기 때문에  
**프론트엔드와 백엔드(API)** 를 **한 프로젝트 안에서 동시에** 다룰 수 있습니다.  

---

## 🚀 내가 Next.js를 선택한 이유

저는 **API가 필요했기 때문에 Next.js를 사용**했습니다.
그 외에 제가 직접 느꼈고, 찾아보면서 알게 된 Next.js의 장점들은 다음과 같습니다:

- ✅ **개인 프로젝트에 API가 필요했기 때문** 
- ✅ **파일 기반 라우팅이 매우 직관적** (폴더 = URL 구조)
- ✅ **SSR/CSR 선택이 자유로움**
- ✅ **SEO 친화적** (페이지가 사전 렌더링되어 크롤링에 유리함)
- ✅ **서버와 클라이언트를 분리해서 로직 구성 가능**

❓ **"SSR/CSR 선택이 자유로움"이란?**

| 용어                              | 의미                                       |
| ------------------------------- | ---------------------------------------- |
| **SSR (Server-Side Rendering)** | 서버에서 HTML을 렌더링해서 보내주는 방식                 |
| **CSR (Client-Side Rendering)** | 브라우저(클라이언트)에서 JS로 렌더링하는 방식 (React 기본 방식) |

- React는?
기본적으로 CSR만 가능해요.
즉, 사용자 브라우저에서 JS로 페이지가 렌더링됩니다.
- Next.js는?
페이지마다 선택적으로 SSR 또는 CSR을 설정할 수 있습니다.
예시: getServerSideProps를 쓰면 SSR, useEffect만 쓰면 CSR
📌 즉, **"이 페이지는 서버에서 먼저 그려줘" / "이건 클라이언트에서 그릴게"**를 개별 페이지마다 선택 가능합니다.

❓ **"SEO 친화적"이란?**
**SEO(Search Engine Optimization): 검색엔진 최적화**
- CSR (React 기본 방식)의 문제점
React는 초기엔 HTML이 빈 상태로 로딩됨
검색엔진 봇이 읽기 힘듦
→ 구글 검색에 잘 안 걸림
- SSR을 쓰면?
Next.js는 서버에서 HTML이 완성된 상태로 응답을 보냄
검색엔진이 내용을 바로 읽을 수 있음
→ 검색 노출에 유리함 (SEO 효과 있음)

## 🔍 React와 Next.js의 주요 차이점

| 항목 | React | Next.js |
|------|--------|----------|
| 구조 | 라이브러리 (UI만 담당) | 프레임워크 (화면 + 서버까지) |
| 라우팅 | 직접 설정 (`react-router-dom`) | **파일 기반 자동 라우팅** |
| 서버 기능 | ❌ 없음 | ✅ API Routes 내장 |
| 데이터 페칭 | 클라이언트 중심 (useEffect 등) | 서버 중심 fetch (`getServerSideProps`, `route.ts`, etc.) |
| 번들링 | 클라이언트 전용 | 클라이언트 + 서버 번들링 |
| SEO | 별도 설정 필요 | 기본적으로 SSR 지원 (SEO 유리) |

## 🧠 핵심 개념: 클라이언트와 서버의 분리

Next.js App Router를 쓸 때 제일 큰 특징 중 하나는 바로 서버 컴포넌트와 클라이언트 컴포넌트를 명확하게 구분한다는 점입니다.

### 👉 `use client` 디렉티브

```tsx
"use client";
```

이걸 써야만 브라우저에서 실행되는 클라이언트 컴포넌트가 됩니다.
이게 없으면 기본적으로 서버 컴포넌트로 동작합니다.


### 👉 라우팅 관련 Hook도 다름

| 용도                  | Hook                                   | 위치                   |
| ------------------- | -------------------------------------- | -------------------- |
| 클라이언트에서 페이지 이동      | `useRouter()` (from `next/navigation`) | 클라이언트 컴포넌트           |
| 쿼리스트링 조회            | `useSearchParams()`                    | 클라이언트 컴포넌트           |
| 서버에서 동적 라우팅 파라미터 사용 | `params` 객체 사용 (서버 파일에서 받음)            | 서버 컴포넌트 (`route.ts`) |

### ⚠️ 사전 지식 안내
이 글은 React에 대한 기본적인 이해가 있으신 분들을 대상으로 작성합니다.
React 사용법에 익숙하지 않으신 분들은 먼저 React 공식 문서나 기본 튜토리얼을 참고하신 후 읽으시면 더 도움이 될 것 같습니다.


## ✨ 마치며
저는 지금까지 React를 잘 사용해왔고,
React가 화면 개발에는 정말 잘 만든 훌륭한 라이브러리라고 생각합니다.
사실 UI를 만드는 데는 React만으로도 충분하다고 생각합니다.

하지만 웹 애플리케이션을 완성하기 위해서는
라우팅, API, 서버 로직 등 좀 더 다양한 기능이 필요한데,
이 부분에서는 Next.js가 훨씬 편리하고 강력하다고 느꼈습니다.

처음에는 Next.js가 다소 생소하게 느껴졌지만,
특히 파일 기반 라우팅 방식이 굉장히 직관적이고 편리하다는 점에서 큰 매력을 느꼈습니다.
또한 API를 같은 프로젝트 내에서 쉽게 구현할 수 있다는 점도 큰 장점이었어요.

그래서 Next.js로 넘어왔고,
