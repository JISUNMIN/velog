## 🧩 SSH (Secure Shell)란?

**SSH(Secure Shell)** 은  
원격 서버에 **보안된 방식으로 접속하기 위한 프로토콜**입니다.  
즉, 암호화된 통신을 통해 서버에 안전하게 접속하거나 데이터를 주고받을 수 있게 합니다.

GitHub나 GitLab에서도 이 SSH를 이용해  
비밀번호 없이 안전하게 인증할 수 있습니다.

---

### 🗝️ SSH Key의 구성

SSH 인증에는 두 개의 키가 사용됩니다.

| 종류 | 설명 |
|------|------|
| **Private Key (개인키)** | 내 컴퓨터에만 저장되는 비밀 키입니다. 절대 외부에 노출하면 안 됩니다. |
| **Public Key (공개키)** | 서버(GitHub, GitLab 등)에 등록하는 키입니다. 누구나 볼 수 있습니다. |

SSH 인증 과정은 이렇게 작동합니다 👇

1. 내 로컬 컴퓨터 → Private Key 보유
2. GitHub 서버 → Public Key 보유
3. 접속 요청 시 서버가 암호화된 메시지를 전송
4. 내 Private Key로 복호화에 성공하면 '인증 완료'


즉, **비밀번호 없이도 내가 ‘인증된 사용자’임을 증명**할 수 있습니다.

---

## 🧠 GPG (GNU Privacy Guard) Key란?

**GPG Key**는 SSH와 비슷하지만,  
Git 커밋의 **신뢰성(“이 커밋이 진짜 이 사람이 한 것인가?”)** 을 보장하기 위해 사용됩니다.

- GPG는 커밋에 **전자서명(Signature)** 을 추가합니다.  
- 이를 통해 다른 사람이 가짜 이름으로 커밋을 남겨도 쉽게 구분할 수 있습니다.

---

### 🔏 GPG Key 사용 예시

```bash
# GPG 키 생성
gpg --full-generate-key

# 생성된 GPG 키 확인
gpg --list-secret-keys --keyid-format=long

# Git 설정에 GPG 키 등록
git config --global user.signingkey <키ID>
git config --global commit.gpgsign true
```

GitHub에서는 GPG로 서명된 커밋에
✅ Verified 표시가 뜹니다.

# 🌐 Git Clone의 두 가지 방식

Git 저장소를 클론할 때는 **HTTPS 방식**과 **SSH 방식** 두 가지가 있습니다.

---

## 1️⃣ HTTPS 형식

```bash
git clone https://github.com/username/repository.git
```

| 항목        | 설명                                          |
| --------- | ------------------------------------------- |
| **인증 방법** | 아이디·비밀번호 or Personal Access Token (PAT)     |
| **보안 수준** | SSL 기반 (안전하지만 원래는 매번 인증 필요)                 |
| **장점**    | 설정이 간단함 (SSH 키 필요 없음)                       |
| **단점**    | 원래는 푸시할 때마다 인증을 요구함 (단, 캐시 덕분에 요즘은 자동 로그인됨) |
| **추천 상황** | 단발성 클론, 임시 접근, SSH 설정이 안 된 환경               |


## 2️⃣ SSH 형식
```bash
git clone git@github.com:username/repository.git
```

| 항목        | 설명                              |
| --------- | ------------------------------- |
| **인증 방법** | SSH Key 기반 (Public/Private Key) |
| **보안 수준** | 키 쌍 기반 암호화로 매우 안전               |
| **장점**    | 한 번 키를 등록하면 이후엔 비밀번호 없이 사용 가능   |
| **단점**    | 최초 설정(SSH Key 생성 및 등록)이 필요      |
| **추천 상황** | 자주 푸시/풀 하는 개인 혹은 팀 프로젝트         |

---

## ⚖️ HTTPS vs SSH 비교 요약
| 구분         | HTTPS                          | SSH                      |
| ---------- | ------------------------------ | ------------------------ |
| **인증 방식**  | 계정 정보 또는 Personal Access Token | SSH Key (Public/Private) |
| **보안성**    | 중간 (SSL 암호화)                   | 매우 높음 (비대칭 키 암호화)        |
| **설정 난이도** | 쉬움                             | 초기에 약간 복잡                |
| **편의성**    | 요즘은 캐싱 덕분에 편리함                 | 한 번 등록 후 완전 자동화          |
| **사용 추천**  | 1회성 접근, 임시 사용                  | 지속적인 협업, 장기 프로젝트         |

---


## 💡 HTTPS로 클론했는데 왜 매번 인증을 안 물어보는가?

원래 HTTPS 방식은 매번 인증(아이디/비밀번호 또는 토큰)을 요청했습니다.
하지만 현재는 Git Credential Manager(GCM) 이라는 시스템이
인증 정보를 OS에 안전하게 저장해두기 때문에,
한 번 입력하면 다시 물어보지 않습니다.

## 🔐 SSH는 컴퓨터가 바뀌면 왜 인증이 안 될까?

SSH 인증은 컴퓨터에 저장된 Private Key 를 기준으로 인증합니다.
즉, 새 컴퓨터에는 Private Key가 없기 때문에,
GitHub가 “이 사용자가 맞다”는 증거를 찾지 못해 인증이 실패합니다.

---

## 🧱 HTTPS vs SSH 인증 구조 요약

| 항목            | HTTPS                 | SSH           |
| ------------- | --------------------- | ------------- |
| **인증 저장 위치**  | OS Credential Manager | 로컬 `.ssh` 폴더  |
| **컴퓨터 바뀔 때**  | 토큰/자격 복사 시 유지 가능      | 새 키 등록 필요     |
| **보안 강도**     | 중간                    | 매우 높음         |
| **초기 설정**     | 간단                    | 약간 복잡         |
| **장기 사용 편의성** | 좋음 (캐싱 덕분)            | 최고 (완전 자동 인증) |

---

## ✅ 정리

SSH Key → 비밀번호 없이 GitHub에 안전하게 인증할 수 있는 열쇠

GPG Key → 커밋에 “이 커밋은 진짜 내가 한 것”임을 증명하는 전자서명

HTTPS Clone → 간단하지만 캐싱 덕분에 인증을 자동으로 처리함

SSH Clone → 한 번 설정하면 비밀번호 없이 안정적으로 사용 가능

---

## 💬 한 줄 요약

🔑 SSH는 내 컴퓨터의 ‘보안 열쇠’, GPG는 커밋의 ‘신뢰 서명’입니다.
HTTPS는 간편하지만 OS가 인증을 캐싱해주는 방식이며,
SSH는 컴퓨터에 저장된 키로만 인증되므로, 새 환경에서는 키 등록이 다시 필요합니다.
