<p>앞선 1~3장에서는 프론트엔드 개발 환경을 구성하는 다양한 도구들(번들러, 트랜스파일러, 빌드 도구)를 각각 따로 살펴보았습니다.</p>
<p>하지만 실제 개발에서는 한 도구가 여러 역할을 동시에 수행하는 경우가 많습니다.</p>
<ul>
<li><strong>Vite</strong> → dev 서버 + 번들러 + 일부 트랜스파일 기능  </li>
<li><strong>esbuild</strong> → 트랜스파일러 + 번들러  </li>
<li><strong>Webpack</strong> → 번들러 + 빌드 도구  </li>
<li><strong>Parcel</strong> → 번들링 + 빌드 + 트랜스파일까지 자동 처리  </li>
</ul>
<p>이처럼 도구의 역할이 겹치기 때문에 헷갈리기 쉽습니다.</p>
<p>그래서 이 장에서는 지금까지 다룬 내용을 기반으로</p>
<blockquote>
<p><strong>“프론트엔드에서 사용되는 주요 도구들은 실제로 어떤 역할을 겸하고 있는가?”</strong><br />를 한 번에 정리해보겠습니다.</p>
</blockquote>
<hr />
<h2 id="💡-빌드-도구는-어떻게-분류해야-할까">💡 빌드 도구는 어떻게 분류해야 할까?</h2>
<p>프론트엔드 도구를 단순히 ‘번들러냐 트랜스파일러냐’로 나누면 오히려 더 헷갈릴 수 있습니다.<br />그래서 이 글에서는 <strong>도구가 프로젝트에서 어떤 레벨을 담당하는지(상위/하위)</strong>와  
<strong>역할이 단일인지/복합인지</strong> 두 가지 기준으로 구분합니다.</p>
<hr />
<h2 id="🥇-1-상위-빌드-도구-vs-하위-빌드-기능-도구">🥇 1) 상위 빌드 도구 vs 하위 빌드 기능 도구</h2>
<h3 id="✔-상위-빌드-도구high-level-build-tools">✔ 상위 빌드 도구(High-level Build Tools)</h3>
<p>→ <strong>프로젝트 전체 빌드 프로세스를 책임지는 도구</strong></p>
<p>특징:</p>
<ul>
<li>dev 서버 + 번들링 + 프로덕션 빌드를 자체적으로 수행  </li>
<li>최종 산출물(dist)을 직접 만들어냄  </li>
<li>플러그인/로더 시스템을 통한 확장성  </li>
<li>프로젝트의 “메인 빌드 시스템”으로 사용됨  </li>
</ul>
<p><strong>예:</strong> Webpack, Vite, Parcel, Rollup</p>
<hr />
<h3 id="✔-하위-빌드-기능-도구low-level-build-tools">✔ 하위 빌드 기능 도구(Low-level Build Tools)</h3>
<p>→ <strong>빌드 과정의 특정 기능(코드 변환, 타입 검사 등)을 담당하는 도구</strong></p>
<p>특징:</p>
<ul>
<li>단독으로 전체 빌드를 구성할 수 없음  </li>
<li>상위 빌드 도구 내부에서 엔진처럼 활용됨  <ul>
<li>Vite → esbuild  </li>
<li>Webpack → Babel / tsc  </li>
</ul>
</li>
<li>주 역할은 트랜스파일, 타입 검사, 다운레벨링 등  </li>
</ul>
<p><strong>예:</strong> tsc, Babel, esbuild</p>
<p>→ esbuild는 번들러 기능을 갖고 있지만 실제로는 상위 도구의 내부 엔진으로 사용하는 경우가 많기 때문에 “하위 도구”로 분류됩니다.</p>
<hr />
<h2 id="🥈-2-단일-역할-도구-vs-다중-역할-도구">🥈 2) 단일 역할 도구 vs 다중 역할 도구</h2>
<h3 id="✔-단일-역할-도구">✔ 단일 역할 도구</h3>
<ul>
<li><strong>tsc</strong> → 트랜스파일 + 타입 검사  </li>
<li><strong>Rollup</strong> → 번들러(트랜스파일은 플러그인 필요)</li>
</ul>
<h3 id="✔-다중-역할-도구">✔ 다중 역할 도구</h3>
<ul>
<li><strong>Vite</strong> → dev 서버 + 번들러 + 트랜스파일 기능  </li>
<li><strong>Webpack</strong> → 빌드 도구 + 번들러  </li>
<li><strong>Parcel</strong> → 빌드 도구 + 번들러 + 트랜스파일  </li>
<li><strong>esbuild</strong> → 트랜스파일러 + 번들러  </li>
</ul>
<hr />
<h2 id="용어-정리">용어 정리</h2>
<h3 id="트랜스파일러transpiler"><strong>트랜스파일러(Transpiler)</strong></h3>
<p>→ TS, JSX, 최신 JS 등 <strong>사람이 쓰기 좋은 코드 → 브라우저가 이해할 수 있는 JS</strong>로 변환<br />예: tsc, Babel, esbuild</p>
<hr />
<h3 id="번들러bundler"><strong>번들러(Bundler)</strong></h3>
<p>→ 여러 JS/CSS/이미지를 <strong>하나의 파일 또는 여러 청크로 묶는 도구</strong><br />예: Webpack, Rollup, esbuild, Parcel</p>
<hr />
<h3 id="빌드-도구build-tool"><strong>빌드 도구(Build Tool)</strong></h3>
<p>→ dev 서버 + 번들링 + 최적화 + 프로덕션 빌드를 총괄하는 상위 도구<br />예: Webpack, Vite, Parcel</p>
<blockquote>
<p><strong>트랜스파일링 → 번들링 → 최종 빌드</strong>라는 흐름은 개념적 모델일 뿐, 실제 빌드 도구들은 이 과정을 재구성해 섞어서 실행합니다.</p>
</blockquote>
<pre><code class="language-text">빌드(Build)
├─ 번들링(bundling)         → 여러 JS 모듈을 하나/소수의 번들로 묶기  (번들러 역할)
├─ 트랜스파일(transpile)    → TS/JSX → JS 변환                         (트랜스파일러 역할)
├─ 다운레벨(downlevel)      → 최신 JS → 구형 브라우저 호환 코드로 변환 (트랜스파일러 역할)
├─ 최적화(optimize)         
│     ├─ minify             → 코드 압축                                (빌드 기능)
│     ├─ tree-shaking       → 사용되지 않는 코드 제거                   (번들러/빌드 기능)
│     └─ code-split         → 필요한 시점에만 파일 로딩                 (번들러 역할)
├─ 리소스 처리(asset)       
│     ├─ CSS 처리/추출  
│     ├─ 이미지/폰트 복사 + 해시 추가                                 (빌드 기능)
│     └─ 캐싱 전략 적용
└─ dist 결과물 생성         → 배포 가능한 최종 폴더 생성                (빌드 도구 기능)</code></pre>
<hr />
<h2 id="🥇-상위-빌드-도구-high-level-build-tools">🥇 상위 빌드 도구 (High-level Build Tools)</h2>
<h3 id="webpack">Webpack</h3>
<p><strong>역할:</strong> 번들러 + 빌드 도구  </p>
<ul>
<li>트랜스파일 기능은 로더(ts-loader, babel-loader)에 의존  </li>
<li>플러그인 생태계 풍부, 설정 자유도 매우 높음  </li>
</ul>
<p><strong>키워드:</strong> <code>번들링 중심</code>, <code>설정 기반</code>, <code>Babel과 찰떡궁합</code></p>
<hr />
<h3 id="vite">Vite</h3>
<p><strong>역할:</strong> 빌드 도구 + dev 서버 + 번들러(Rollup 기반)  </p>
<ul>
<li><strong>dev 환경:</strong> esbuild로 초고속 변환 + <strong>HMR 지원 dev 서버 제공</strong>  <ul>
<li>번들링 없이 ESM으로 제공 → 빠른 서버 재시작  </li>
<li>변경된 모듈만 교체하는 HMR로 빠른 피드백  </li>
</ul>
</li>
<li><strong>build 환경:</strong> Rollup 기반으로 프로덕션 최적화 번들 생성  </li>
</ul>
<p><strong>키워드:</strong> <code>빠름</code>, <code>Rollup 기반</code>, <code>개발 경험 최고</code>, <code>ESM 기반 dev 서버</code>, <code>HMR</code></p>
<hr />
<h3 id="rollup">Rollup</h3>
<p><strong>역할:</strong> 번들러</p>
<ul>
<li>라이브러리 번들링에 최적  </li>
<li>출력 형태 다양(ESM, CJS 등)  </li>
<li>Tree-shaking 강력 <ul>
<li>** Rollup은 플러그인 시스템이 매우 강력하여
TypeScript, Babel, terser(압축), CSS 처리 등 다양한 플러그인을 붙이면<br />사실상 “빌드 도구” 수준의 전체 파이프라인을 구성할 수 있음**  </li>
</ul>
</li>
</ul>
<p><strong>키워드:</strong> <code>라이브러리 제작용</code>, <code>정적 분석</code>, <code>가벼운 번들</code></p>
<hr />
<h3 id="parcel">Parcel</h3>
<p><strong>역할:</strong> 번들러 + 빌드 도구 + 자동 트랜스파일 내장  </p>
<ul>
<li>설정 없이 바로 사용 가능(Zero-config)  </li>
<li>TS, JSX, 이미지, 스타일 자동 처리  </li>
<li>초심자 친화적  </li>
</ul>
<p><strong>키워드:</strong> <code>Zero-config</code>, <code>자동화</code>, <code>초심자 친화적</code></p>
<hr />
<h2 id="하위-빌드-기능-도구-low-level-build-tools">하위 빌드 기능 도구 (Low-level Build Tools)</h2>
<h3 id="typescript-compiler-tsc">TypeScript Compiler (tsc)</h3>
<p><strong>역할:</strong> TS → JS 변환 + 타입 검사  </p>
<ul>
<li>번들링 없음  </li>
<li>다운레벨링 약함  </li>
</ul>
<p><strong>키워드:</strong> <code>타입 검사</code>, <code>표준 TS 컴파일러</code></p>
<hr />
<h3 id="babel">Babel</h3>
<p><strong>역할:</strong> TS/JSX 트랜스파일 + 브라우저 호환성 변환  </p>
<ul>
<li>ESNext → ES5 다운레벨링 최강  </li>
<li>Webpack과 궁합 좋음  </li>
</ul>
<p><strong>키워드:</strong> <code>호환성 최강</code>, <code>플러그인 생태계 풍부</code></p>
<hr />
<h3 id="esbuild">esbuild</h3>
<p><strong>역할:</strong> 트랜스파일러 + 경량 번들러  </p>
<ul>
<li>dev 환경에서 가장 빠른 트랜스파일링  </li>
<li>Vite dev 서버의 핵심 엔진  </li>
<li>다운레벨링은 Babel보다 단순  </li>
<li><strong>실무에서는 보통 메인 빌드 도구로 사용되지 않고
Vite·Turbopack 같은 상위 도구 내부의 빌드 엔진으로 동작</strong></li>
</ul>
<p><strong>키워드:</strong> <code>초고속</code>, <code>Go 기반</code>, <code>빌드 최적화</code></p>

<h2 id="📌-한-줄-요약">📌 한 줄 요약</h2>
<blockquote>
<p><strong>프론트엔드 빌드 도구는 역할 개수보다 “어떤 레벨(상위/하위)을 담당하는가”가 더 중요하며, 대부분의 최신 도구는 여러 기능을 동시에 수행하는 멀티툴이다.</strong></p>
</blockquote>
